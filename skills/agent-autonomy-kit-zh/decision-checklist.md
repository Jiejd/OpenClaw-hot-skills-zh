# 决策检查清单

**目的：** 在存在自主执行路径时，防止请求人类许可  
**何时使用：** 在向人类询问"我该做什么？"或"下一步是什么？"之前  
**状态：** 在任务选择时请求人类输入前的**必须**通过的关卡  
**版本：** 1.0  
**最后更新：** 2026 年 2 月 4 日

---

## 问题所在

智能体在以下情况下仍然向人类寻求下一步指导：
- 队列中有 HIGH 优先级任务等待执行
- 计划/日程已经记录了下一步操作
- 5D 循环阶段有明确的下一步
- 不存在真正的阻塞

**结果：** 自主执行在心跳期间正常工作，但在决策边界处失败。

---

## 解决方案

**在询问"我接下来该做什么？"之前，运行此检查清单。**

如果你通过了所有检查 → 你确实被阻塞了，可以询问。

如果你未通过任何检查 → 你并未被阻塞，你只是还没有自主执行而已。

## 检查清单

### ✅ 检查 1：这在队列中吗？

**操作：**
```bash
node skills/agent-autonomy-kit/check-queue.js
```

**判断：**
- 退出码 1（存在 HIGH/CRITICAL 任务）：**停止。不要问。执行队列任务。**
- 退出码 0（无 HIGH 任务）：继续检查 2

**原因：**  
队列是优先级工作的权威来源。如果存在 HIGH 优先级任务，你不需要许可——你已经有方向了。

**失败示例：**
```
智能体："我们接下来该处理什么？"
队列：14 个 HIGH 优先级任务
结果：智能体应该为最高优先级任务 spawn，而不是询问
```

**通过示例：**
```
智能体："卡住了：需要部署用的 API 密钥"
队列：部署到生产环境是 HIGH 优先级
结果：合理的阻塞，可以询问
```

---

### ✅ 检查 2：这在计划/日程中吗？

**操作：**
```bash
# 检查心跳状态中的计划任务
cat memory/heartbeat-state.json | jq '.pending'
```

**判断：**
- 如果未来 24 小时内有计划任务：**停止。不要问。执行日程。**
- 如果没有计划任务：继续检查 3

**原因：**  
计划记录了已经做出的决定。如果日程写着"2 月 6 日上午 9 点发布"，你不需要审批——你需要执行。

**失败示例：**
```
智能体："内容已就绪。我们应该按计划明天发布吗？"
日程："Observability 发布 2 月 6 日上午 9 点 PST"
当前时间：2 月 5 日下午 6:20 PST
结果：计划已存在，智能体应执行日程，而不是询问
```

**通过示例：**
```
智能体："计划明天发布，但发现了关键 bug。应该延期吗？"
日程：发布已计划
新情况：关键 bug（计划外的阻塞）
结果：合理的问题——需要战略决策
```

---

### ✅ 检查 3：我检查了 5D 循环阶段吗？

**操作：** 读取 HEARTBEAT.md 并检查每个阶段：

1. **DISCOVER（发现）：** 上次发现运行是什么时候？
   - 检查：`memory/heartbeat-state.json` → `lastDiscovery`
   - 如果超过 2 小时：**停止。不要问。Spawn Scout。**

2. **DEFINE（定义）：** 是否有未分类的机会？
   - 检查：`process/OPPORTUNITIES.md` 中的新发现
   - 如果有：**停止。不要问。Spawn Sage。**

3. **DESIGN（设计）：** 是否有缺少设计的规格？
   - 检查：`tasks/DEFINE-*.md` 文件是否需要设计
   - 如果有：**停止。不要问。Spawn Pixel。**

4. **DEVELOP（开发）：** 是否有已设计好待构建的项目？
   - 检查：有设计但没有代码的项目文件夹
   - 如果有：**停止。不要问。Spawn Link。**

5. **DEPLOY（部署）：** 是否有已构建但未发布的项目？
   - 检查：已完成但未在分发渠道中的项目
   - 如果有：**停止。不要问。Spawn Spark/Echo。**

**判断：**
- 如果任何阶段显示有工作要做：**停止。不要问。Spawn 对应智能体。**
- 如果所有阶段都是最新的：继续检查 4

**原因：**  
5D 循环能自主生成工作。如果阶段过时，你有方向——你不需要许可。

**失败示例：**
```
智能体："队列已清空。我该做什么？"
发现阶段：上次运行 4 小时前（过时）
结果：智能体应该 spawn Scout，而不是询问
```

**通过示例：**
```
智能体："所有 5D 阶段最新，队列清空，无计划任务。准备接收新方向。"
5D 循环：所有阶段健康（<2 小时）
队列：空
日程：未来 24 小时无安排
结果：确实空闲，合理的问题
```

### ✅ 检查 4：我真的被阻塞了吗？

**自问以下问题：**

1. **缺少我找不到的信息？**
   - 搜索过 .secrets/ 中的凭证？ ✅
   - 查阅过文档？ ✅
   - 尝试过网页搜索？ ✅
   - 仍然缺少关键信息？ → 可以询问

2. **等待外部依赖？**
   - 需要人类审批（部署到生产环境、预算）？ → 可以询问
   - 需要 API 访问/凭证？ → 可以询问
   - 第三方服务宕机？ → 应升级，而非请求方向

3. **遇到无法解决的技术阻塞？**
   - 尝试过调试？ ✅
   - 搜索过错误信息？ ✅
   - 查阅过 GitHub issues？ ✅
   - 仍然卡住？ → 可以询问

**判断：**
- 如果确实被阻塞：继续检查 5
- 如果未被阻塞：**停止。不要问。去做工作。**

**原因：**  
"我不知道该做什么"通常是"我没有检查队列/日程/5D 循环"。真正的阻塞是具体的（"缺少 X 的 API 密钥"）。

**失败示例：**
```
智能体："任务完成了。下一步呢？"
实际情况：未被阻塞，只是没有检查队列
结果：先运行队列检查器
```

**通过示例：**
```
智能体："需要生产部署审批以修复安全补丁"
实际情况：被策略阻塞（需要人类审批）
结果：合理的升级请求
```

### ✅ 检查 5：这是否是超出我职责范围的战略决策？

**战略决策 = 人类权限：**
- 产品方向变更（调整功能、淘汰产品）
- 预算分配（订阅付费服务、雇佣外包）
- 策略变更（更改部署流程、访问控制）
- 对外承诺（会议演讲、合作伙伴关系）

**战术决策 = 智能体权限：**
- 下一个队列任务做什么（队列优先级决定）
- 如何实现一个功能（技术决策）
- 何时 spawn subagent（工作流优化）
- 内容排期（如果计划已存在）

**判断：**
- 如果是战略决策：**可以询问**（这是例外情况）
- 如果是战术决策：**停止。不要问。自主决定。**

**原因：**  
智能体应该拥有执行权。人类应该拥有战略权。不要将战术决策升级。

**通过示例：**
```
智能体："我们应该将 forAgents.dev 转向专注于 MCP 工具而不是智能体吗？"
类型：战略决策（产品方向）
结果：合理的问题——需要人类决策
```

**失败示例：**
```
智能体："我应该用 markdown 还是 HTML 写博客文章？"
类型：战术决策（实现细节）
结果：智能体应该自己决定（markdown 是标准）
```

## 如何使用此检查清单

### 方式 1：手动检查清单（始终可用）

在询问"我接下来该做什么？"之前，逐步执行检查 1-5。如果你通过了所有检查，说明你确实被阻塞——可以询问。

### 方式 2：自动检查（推荐）

使用 `--pre-response` 标志运行队列检查器：

```bash
node skills/agent-autonomy-kit/check-queue.js --pre-response
```

这会自动检查：
- 队列状态（检查 1）
- heartbeat-state.json 中的计划任务（检查 2）
- 过时的 5D 阶段（检查 3）

然后手动验证检查 4-5（阻塞 vs 战略）。

## 衡量成功

### 使用检查清单之前
- "我们接下来该处理什么？" = 常见
- 请求许可 = 任务完成后的默认行为
- 自主执行 = 40%（仅心跳入口）

### 使用检查清单之后
- "我们接下来该处理什么？" = 罕见（仅当确实空闲时）
- 请求许可 = 仅用于阻塞/战略决策
- 自主执行 = 95%+（所有决策边界）

### 关键指标

在每日日志中追踪：
- **每次会话的许可请求数：** 目标 <1（仅真正阻塞时）
- **询问前检查队列的比例：** 目标 100%
- **战略 vs 战术请求：** 战略可以，战术应为 0

## 常见问题

### 问：这感觉步骤很多。我真的需要所有 5 个检查吗？

**答：** 不一定。通常检查 1（队列）就足够了。但如果你准备向人类请求方向指导，花 2 分钟做检查清单能节省所有人的时间。

**快捷方式：** 运行 `check-queue.js --pre-response`（覆盖检查 1-3）。

### 问：如果我有 90% 的把握应该询问，但某个检查未通过怎么办？

**答：** 遵循检查结果。这就是重点——我们信任系统而非直觉。如果队列说有 HIGH 任务，就做那个任务。不要问。

### 问：我可以对任务提问以澄清吗？

**答：** 可以！澄清问题和请求许可是不同的：

**请求许可（不要问）：**
- "我们该做什么？"
- "我们应该发布吗？"

**澄清问题（可以问）：**
- "任务说'修复 auth bug'——这是登录问题还是 OAuth 问题？"
- "HIGH 优先级：部署。生产部署前需要审批吗？"

### 问：如果队列中只有 LOW 优先级任务怎么办？

**答：** 检查 5D 循环阶段。如果阶段空闲且无计划工作，你可以：
1. 选择 LOW 任务并执行
2. 做主动发现
3. 向人类请求新的优先级

LOW 任务不阻止询问——只有 HIGH/CRITICAL 才会。

### 问：如果我不同意队列优先级怎么办？

**答：** 不要静默覆盖队列。而是：
1. 按队列原样执行（做 HIGH 任务）
2. 之后记录为什么优先级可能有误
3. 向人类或 Rhythm（队列负责人）建议重新排序

除非你有数据证明系统有误，否则信任系统。
