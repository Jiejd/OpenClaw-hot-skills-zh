# 自主化工具包故障诊断

**日期：** 2026 年 2 月 4 日  
**分析员：** Subagent (harmony-autonomy-diagnosis)  
**状态：** 🔴 严重 — 系统在心跳期间强制执行自主化，但在任务边界处失败

---

## 概要

Autonomy Kit 在**心跳期间**成功强制执行了自主执行，但在**任务完成边界**处完全失败。完成任务后，智能体退回到请求许可（"我们接下来该处理什么？"、"你想什么时候发布？"），而不是查询队列。

**根本原因：** 缺少 post-task protocol（任务后协议）。队列检查器仅在心跳入口运行，留下了一个缺口——智能体完成工作后立即转向人类而不是循环回到队列。

**影响：** 自主执行在 40% 的时间里有效（心跳期间）但在 60% 的时间里失败（任务边界处），造成了工具包"不起作用"的印象。

## 根本原因分析

### 缺口 #1：缺少 Post-Task Protocol

**当前流程：**
```
1. 心跳触发
2. 运行队列检查器（必须执行）
3. 为 HIGH 任务 spawn 智能体
4. 智能体完成任务
5. ???（未定义行为）
```

**第 5 步就是问题所在。** 没有关于以下内容的协议：
- 智能体完成任务后立即做什么
- 是否需要再次检查队列
- 何时将控制权交还给人类 vs 继续工作

**结果：** 智能体退回到默认 LLM 行为（向人类请求下一步指令）。

### 缺口 #2：队列检查器范围太窄

**当前范围：** 仅入口（心跳开始）

**缺失范围：**
- 任务完成边界
- Subagent 返回点
- 审查通过点
- 任何需要回答"下一步是什么？"的时刻

**队列检查器在一个检查点强制执行自主化，但智能体需要在每个检查点都有自主化能力。**

### 缺口 #3：HEARTBEAT.md 表述不清

**当前 HEARTBEAT.md（第 42-43 行）：**
```markdown
## 心跳执行

1. **人类消息优先** — 始终立即处理直接请求
2. **队列强制执行（必须）** — 在做任何其他事之前运行队列检查器
```

**它说了什么：** 在心跳开始时检查队列。

**它没说什么：**
- 完成任务后检查队列
- 在向人类请求输入之前检查队列
- 当 subagent 返回结果时检查队列

**指导是正确的但不完整。** 它处理了入口但没有处理回路。

### 缺口 #4：与人类交互前缺少"决策检查清单"

目前，智能体可以自由地：
- 询问"我们接下来该处理什么？"
- 说"你觉得 X 怎么样？"
- 请求决策审批

**缺失：** 一个预响应关卡，要求回答：
1. 队列中有工作吗？
2. 我查询过队列了吗？
3. 我是因为被阻塞而询问，还是出于习惯而询问？

## 具体失败模式

### 失败模式 1：任务完成 → 立即询问

**场景：**
```
15:52 - Colony Skill 任务完成
15:52 - 队列已更新（移到 Done）
15:53 - 智能体发送："任务完成！我们接下来该处理什么？"
```

**为什么失败：** 任务完成后没有回路到队列检查器。

**预期行为：**
```
15:52 - Colony Skill 任务完成
15:52 - 队列已更新（移到 Done）
15:52 - 再次运行队列检查器
15:52 - 退出码 1（仍有 14 个 HIGH 优先级任务）
15:53 - 为下一个 HIGH 优先级任务 spawn 智能体
```

### 失败模式 2：Subagent 返回 → 推给人类

**场景：**
```
智能体 spawn Echo 撰写活动内容
Echo 完成并报告
主智能体读取 Echo 的报告
主智能体发送："Echo 完成了！你想什么时候发布？"
```

**为什么失败：** Subagent 返回 = 未定义的转换点，默认询问人类。

**预期行为：**
```
智能体 spawn Echo 撰写活动内容
Echo 完成并报告
主智能体读取 Echo 的报告
主智能体运行队列检查器
主智能体检查"发布"是否在队列中为 HIGH 优先级
如果是：执行发布
如果否：添加到队列稍后处理或按计划执行
```

### 失败模式 3：计划执行 → 请求许可

**场景：**
```
智能体有详细的发布计划（第 2 天：2 月 6 日上午 9 点 PST）
当前时间：2 月 5 日下午 6:20 PST
智能体询问："内容已就绪。我们应该按计划明天发布吗？"
```

**为什么失败：** 智能体拥有所有信息（日期、计划、准备状态）但推迟了决策。

**预期行为：**
```
智能体有详细的发布计划
智能体检查：当前时间 >= 计划时间？否。
智能体检查：内容准备好了吗？是。
智能体检查：有阻塞问题吗？否。
智能体结论：按计划执行（明天上午 9 点）
智能体操作：设置提醒，继续其他队列任务
```

### 失败模式 4：队列清空 → 发现抑制

**场景：**
```
队列检查器返回退出码 0（无 HIGH 任务）
智能体发送："队列已清空！HEARTBEAT_OK。"
```

**为什么失败：** 将"无 HIGH 任务"视为"什么都不做"，而不是"可以安全地做主动工作"。

**预期行为（来自 HEARTBEAT.md 第 99-103 行）：**
```
队列检查器返回退出码 0
智能体检查 5D 循环阶段：
- 阶段 1 (DISCOVER)：上次运行 2 小时前 → spawn Scout
- 阶段 2 (DEFINE)：3 个机会需要规格 → spawn Sage
- 阶段 3 (DESIGN)：1 个规格需要设计 → spawn Pixel
智能体 spawn Scout 进行发现
```

## 为什么当前系统允许这种行为

### 1. 队列检查器 = 仅入口关卡

**当前架构：**
```
心跳开始 ──[必须检查]──> 工作
                          │
                          v
                      完成
                          │
                          v
                       ??? <── 无关卡
```

**缺失的部分：**
```
心跳开始 ──[检查 1]──> 工作
                      │
                      v
                   完成
                      │
                      v
                  [检查 2] ──> 循环回来或升级
```

### 2. HEARTBEAT.md = 被动指导

**当前状态：** 包含指令的 Markdown 文件

**问题：** Markdown 不能强制执行。智能体可以：
- 跳过阅读（如果不在工作区上下文中）
- 阅读了但在任务执行期间忘记
- 在任务后做出不同解读

**类比：** 这就像有一个"请勿酒驾"的标语 vs 点火互锁装置。一个是建议，一个是强制执行。

### 3. 没有预响应验证

智能体可以输入回复并发送，而无需任何验证它们是否：
- 查询过队列
- 检查了自主选项
- 穷尽了非人类路径

**在"智能体想问人类"和"消息发送给人类"之间没有关卡。**

### 4. Subagent 边界未定义

当 subagent 完成时：
- 主智能体收到通知
- 主智能体读取 subagent 的输出
- **主智能体没有关于下一步该做什么的协议**

当前行为：将其视为完成自己的工作 → 询问人类。

缺失行为：将其视为心跳唤醒 → 检查队列，检查 5D 循环，自主决策。

### 修复 3：队列检查器扩展

**文件：** `skills/agent-autonomy-kit/check-queue.js`

添加一个**预响应模式**，智能体可以在询问人类之前调用：

```javascript
// 添加到 check-queue.js

function preResponseCheck() {
  // 运行正常队列检查
  const queueStatus = checkQueue();
  
  // 同时检查心跳状态中的计划任务
  const stateFile = path.join(workspaceRoot, 'memory/heartbeat-state.json');
  if (fs.existsSync(stateFile)) {
    const state = JSON.parse(fs.readFileSync(stateFile, 'utf-8'));
    
    // 检查未来 24 小时的计划操作
    if (state.pending && state.pending.length > 0) {
      console.log(`\n⏰ Scheduled tasks: ${state.pending.length}`);
      state.pending.forEach(task => console.log(`   • ${task}`));
      console.log(`\nYou have scheduled work. Execute the schedule, don't ask.`);
    }
    
    // 检查过时的 5D 阶段
    const now = Date.now();
    const lastDiscovery = state.lastDiscovery ? new Date(state.lastDiscovery).getTime() : 0;
    const staleHours = (now - lastDiscovery) / (1000 * 60 * 60);
    
    if (staleHours > 2) {
      console.log(`\n🔍 Discovery is stale (${staleHours.toFixed(1)}h ago)`);
      console.log(`Spawn Scout for discovery before asking human.`);
    }
  }
  
  return queueStatus;
}

// 添加 CLI 标志
if (process.argv.includes('--pre-response')) {
  preResponseCheck();
}
```

**用法：**
```bash

```

这给智能体提供了一个命令来检查：
1. 队列状态
2. 计划任务
3. 过时的 5D 阶段

如果其中任何一个返回工作，智能体不应询问人类。

## 架构评估

### 架构是否合理？

**是的** — 架构从根本上说是正确的：

✅ 基于队列的工作管理  
✅ 通过退出码进行程序化强制执行  
✅ 优先级系统（CRITICAL、HIGH、MEDIUM、LOW）  
✅ 用于生成性工作的 5D 循环  
✅ 清晰的文档

**问题不在于设计，而在于覆盖范围。**

### 缺少什么？

**在每个决策边界的强制执行，而不仅仅是心跳入口。**

当前覆盖：
```
[已强制] 心跳入口
[未强制] 任务完成
[未强制] Subagent 返回
[未强制] 预响应时刻
[未强制] 审查通过点
```

应该是：
```
[已强制] 心跳入口
[已强制] 任务完成
[已强制] Subagent 返回
[已强制] 预响应时刻
[已强制] 审查通过点
```

**每个"下一步是什么？"的时刻都需要队列检查。**

## 对 Ryan 问题的具体回答

### 问：工具包架构是否合理但在我的工作流中实现不好？

**答：** 架构合理。实现存在**覆盖缺口**。

工具包在入口点（心跳开始）正确地强制执行了自主化，但没有在任务边界处强制执行。这不是工作流问题——这是一个**缺失的协议**。

Kai 在心跳期间正确地遵循了 HEARTBEAT.md。Kai 没有关于任务后行为的指导，所以退回到了 LLM 默认行为（询问人类）。

### 问：HEARTBEAT.md 是否应该有更严格的安全保障？

**答：** 是的。添加：

1. **回路指令：** "完成任务后，再次运行队列检查器"
2. **预响应检查：** "在询问人类之前，运行决策检查清单"
3. **显式规则：** "如果队列有答案，你不可以问'下一步是什么？'"

当前 HEARTBEAT.md 90% 正确但缺少回路。

### 问：队列检查器是否应该阻止除 spawn 智能体之外的所有响应？

**答：** 不——太极端了。但应添加**预响应模式**。

**为什么阻止所有响应是错误的：**
- 智能体需要报告完成
- 智能体需要升级真正的阻塞
- 智能体需要询问战略问题

**正确的方法：**
- 添加 `check-queue.js --pre-response` 标志
- 使其在询问"下一步是什么？"类问题之前成为必须项
- 允许智能体询问真正的战略/阻塞问题

### 问：我们是否需要一个在每次响应之前运行的"决策检查清单"？

**答：** 是的——在向人类请求下一步指导之前。

**不是在每次响应之前**（太限制），而是在以下响应之前：
- "我们接下来该做什么？"
- "你觉得 X 怎么样？"
- "我们应该发布吗？"
- "你希望什么时候...？"

这些是应该触发检查清单的请求许可模式。

## 结论

**Autonomy Kit 有效——但仅在入口关卡。**

系统在心跳开始时成功强制执行了队列优先。它在任务完成边界处失败，因为缺少"完成任务后该做什么"的协议。

**根本原因：** 缺少回路指令。智能体在唤醒时检查队列，但在任务完成时不检查队列。

**修复很简单：**
1. 添加任务后协议（再次检查队列）
2. 添加预响应检查清单（除非被阻塞否则不要问）
3. 用 --pre-response 模式增强队列检查器
4. 更新 HEARTBEAT.md 和 AGENTS.md 中的回路规则

**这不是根本性缺陷。** 这是一个覆盖缺口。架构合理。强制执行机制有效。我们只需在更多检查点应用它。

**预计修复时间：** 90 分钟实施 + 48 小时实际使用。

**预期结果：** 除非确实被阻塞，否则零请求许可。每个决策边界的自主执行。
